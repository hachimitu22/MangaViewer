## 正常系

### 対象フォルダに zip が 0 件でも正常終了する

- **前提**
  - 実ファイル I/O で空の対象フォルダを用意する。
  - 対象フォルダ直下には zip を置かず、サブフォルダ配下には zip を配置しておく。
  - ログ出力先（JSONL）を指定する。
- **操作**
  - ImportZipsService を実行する。
- **結果**
  - 異常終了せずに処理が完了する。
  - `zip単位結果一覧=[]` が返る。
  - `全体サマリ` は `対象zip件数=0 / 成功件数=0 / 失敗件数=0` が返る。
  - `実行結果種別=SUCCESS` が返る。
  - サブフォルダ配下の zip は探索対象外として無視される。
  - ログファイルは作成される（ヘッダ不要、必要行のみ）。

---

### zip が複数件ある場合に一部失敗しても他 zip の処理を継続する

- **前提**
  - 実ファイル I/O で対象フォルダに複数 zip を配置する。
  - 少なくとも 1 件は成功可能な zip、1 件は失敗する zip（例: 画像 0 件）を含める。
- **操作**
  - ImportZipsService を実行する。
- **結果**
  - 失敗 zip が発生しても後続 zip の処理を継続する。
  - zip 単位結果一覧に成功・失敗が混在して記録される。
  - 全体サマリの件数が実 zip 件数と一致する。
  - `実行結果種別=PARTIAL_FAILURE` が返る。

---

### 同一入力では zip の処理順が安定する

- **前提**
  - 実ファイル I/O で対象フォルダに、比較順が揺れやすい zip 名を混在配置する。
    - 例: `vol1.zip`, `vol02.zip`, `VOL2.zip`, `ＶＯＬ３.zip`, `vol10.zip`
  - 同一の入力フォルダ内容・設定で 2 回以上実行できる状態にする。
  - ログ出力先（JSONL）を指定する。
- **操作**
  - ImportZipsService を同一入力で複数回実行し、各回の zip 単位ログまたは結果一覧の処理順を取得する。
- **結果**
  - すべての実行回で zip の処理順が一致する（非決定的な順序揺れがない）。
  - 順序は「zip ファイル名の自然順」を満たす。
  - 比較は大文字小文字非依存・ロケール非依存・全角半角差異吸収（NFKC）を前提に行われる。

---

### zip 単位でロールバックされ失敗 zip のゴミファイルが残らない

- **前提**
  - 実ファイル I/O で、展開途中で失敗する zip（例: 上限超過ファイルを含む）を用意する。
  - 成功する別 zip も同時に用意する。
- **操作**
  - ImportZipsService を実行する。
- **結果**
  - 失敗した zip に対応する一時展開物・中間生成物が作業領域に残らない。
  - 失敗 zip に対応するメディア永続化は行われない。
  - 成功 zip 側の永続化結果には影響しない。

---

### ログに zip 名・成否・無視ファイル一覧が出力される

- **前提**
  - 実ファイル I/O で、画像以外ファイル（例: `__MACOSX/`, `Thumbs.db`, `readme.txt`）を含む zip を用意する。
  - ログ出力先（JSONL）を指定する。
- **操作**
  - ImportZipsService を実行する。
- **結果**
  - zip ごとのログ行に zip 名が含まれる。
  - zip ごとのログ行に成否（SUCCESS / FAILED）が含まれる。
  - 無視された非画像ファイル一覧が出力される。
  - 失敗時は理由、成功時は mediaId が出力される。

---

### JSONL ログのキー整合性を満たす

- **前提**
  - 実ファイル I/O で成功 zip と失敗 zip を混在させた入力を用意する。
  - ログ出力先（JSONL）を指定する。
- **操作**
  - ImportZipsService を実行し、出力された JSONL を1行ずつ JSON として読み込む。
- **結果**
  - zip 単位ログの必須キー（`zipName`, `status`, `ignoredFiles`）が常に存在する。
  - `status=SUCCESS` の行は `mediaId` を持ち、`reasonCode`/`reasonDetail` は未設定または空である。
  - `status=FAILED` の行は `reasonCode` を持ち、`mediaId` は未設定である。
  - `status=FAILED` の `reasonCode` は採用列挙値のいずれかに一致する。
  - `reasonDetail` は任意項目とし、存在する場合は補助情報（対象ファイル名など）を含められる。
  - 配列キー `ignoredFiles` は常に配列型である（空配列を許容）。
  - 集計値（成功/失敗件数）が zip 単位ログ件数と矛盾しない。

---


### zipごとの最大画像数の境界値で失敗隔離とロールバックを満たす

- **前提**
  - `zipごとの最大画像数=MAX_IMAGES_PER_ZIP` を設定する。
  - 実ファイル I/O で以下を同一実行に混在させる。
    - ケースA: `MAX_IMAGES_PER_ZIP-1` 枚の zip（成功想定）
    - ケースB: `MAX_IMAGES_PER_ZIP` 枚の zip（成功想定）
    - ケースC: `MAX_IMAGES_PER_ZIP+1` 枚の zip（失敗想定）
  - ログ出力先（JSONL）を指定する。
- **操作**
  - ImportZipsService を実行し、永続化結果・作業領域・JSONL を確認する。
- **結果**
  - ケースA/B は成功し、メディア永続化が完了する。
  - ケースC は失敗し、当該 zip の永続化は行われない（zip 単位原子性）。
  - ケースC に対応する一時展開物・中間生成物は残らない（ロールバック）。
  - ケースC の `reasonCode` は `ZIP_IMAGE_COUNT_LIMIT_EXCEEDED` と完全一致する（`reasonDetail` は任意）。
  - ケースC の失敗があっても他 zip の処理継続性が保たれる。

---

### 画像ファイルの最大バイト数の境界値で reasonCode 分類とログ整合を満たす

- **前提**
  - `画像ファイルの最大バイト数=MAX_IMAGE_BYTES` を設定する。
  - 実ファイル I/O で以下を用意する。
    - ケースA: `MAX_IMAGE_BYTES-1` の画像を含む zip（成功想定）
    - ケースB: `MAX_IMAGE_BYTES` の画像を含む zip（成功想定）
    - ケースC: `MAX_IMAGE_BYTES+1` の画像を含む zip（失敗想定）
  - ログ出力先（JSONL）を指定する。
- **操作**
  - ImportZipsService を実行し、zip 単位結果と JSONL を照合する。
- **結果**
  - ケースA/B は成功し、`status=SUCCESS` と `mediaId` が記録される。
  - ケースC は失敗し、`status=FAILED` と `reasonCode` が記録される（`reasonDetail` は任意）。
  - ケースC の `reasonCode` は `IMAGE_FILE_SIZE_LIMIT_EXCEEDED` と完全一致する。
  - `reasonDetail` は補助文字列レベル（存在/非存在や部分一致）で検証する。
  - ケースC 失敗時もケースA/B の永続化結果は維持される（失敗隔離）。

---

### 1実行あたり最大zip処理件数の境界値で実行全体ポリシーを満たす

- **前提**
  - `1実行あたり最大zip処理件数=MAX_ZIPS_PER_RUN` を設定する。
  - 実ファイル I/O で zip 件数を以下にした入力を用意する。
    - ケースA: `MAX_ZIPS_PER_RUN-1`
    - ケースB: `MAX_ZIPS_PER_RUN`
    - ケースC: `MAX_ZIPS_PER_RUN+1`
  - ログ出力先（JSONL）を指定する。
- **操作**
  - 各ケースで ImportZipsService を実行し、戻り値・永続化結果・JSONL を確認する。
- **結果**
  - ケースA/B は受理され、件数矛盾なく処理される。
  - ケースC は**実行全体を拒否**し、zip 処理を開始しない。
  - `実行結果種別=INVALID_INPUT_RUN_ZIP_COUNT_LIMIT_EXCEEDED` / `zip単位結果一覧=[]` / `全体サマリ=0件` を返す。
  - 拒否時に永続化・作業領域への副作用を残さない。

## medium テスト方針
- small では入力検証や純粋ロジックを担保し、medium では「実ファイル I/O を伴う zip 取り込み」の統合観点を担保する。
- medium では `ImportZipsService` と実際のファイルシステム・永続化・ログ出力の接続点を重視し、zip 単位の継続性と原子性（ロールバック）を確認する。
- JSONL ログは運用監査と障害解析の一次情報となるため、キー整合性と件数整合性を確認対象に含める。
