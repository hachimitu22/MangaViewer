# ImportZipsService テストケース（medium: 実 I/O 統合）

## 目的
- medium は、`ImportZipsService` と **実ファイル I/O（FS）・実トランザクション** の結合点を検証する。
- small と同一観点の重複を避け、1 ケース 1 責務で「どちらで担保するか」を明確化する。
- ログ出力（`AppLogger` による人間可読ログ/JSONL整形・出力）は CLI / infrastructure の責務であり、本書の対象外とする。

## small / medium の責務分担（重複整理）
| 観点 | small で担保 | medium で担保 |
|---|---|---|
| 入力検証（契約違反、上限判定の境界） | 実施（純粋ロジック） | 原則実施しない（I/O 副作用有無の確認が必要な最小ケースのみ） |
| 並び順決定（zip名/画像名の自然順） | 実施（決定ロジック） | 同一入力で処理順が安定し、実行ごとの差異が出ないことを実 I/O で確認 |
| 結果種別集計（SUCCESS/PARTIAL_FAILURE/INVALID_INPUT*） | 実施（集計ロジック） | 実 I/O を伴う複数 zip 実行で集計と実副作用が整合することを確認 |
| reasonCode 決定 | 実施（`NO_IMAGES` 等の分類） | 実 I/O を伴う失敗入力で reasonCode が設計どおり返ることを確認 |
| zip 単位原子性 / ロールバック | 対象外（呼び出し選択のみ） | 実施（失敗 zip の副作用が残らない） |
| ログ出力（人間可読/JSONL） | 対象外 | 対象外（CLI / infrastructure のテストで担保） |

## ケース詳細

### 対象フォルダに zip が 0 件でも正常終了する（実 FS 探索）

- **前提**
  - 実ファイル I/O で空の対象フォルダを用意する。
  - 対象フォルダ直下には zip を置かず、サブフォルダ配下には zip を配置しておく。
- **操作**
  - ImportZipsService を実行する。
- **結果**
  - 異常終了せずに処理が完了する。
  - `zip単位結果一覧=[]` が返る。
  - `全体サマリ` は `対象zip件数=0 / 成功件数=0 / 失敗件数=0` が返る。
  - `実行結果種別=SUCCESS` が返る。
  - サブフォルダ配下の zip は探索対象外として無視される。

---

### zip が複数件ある場合に一部失敗しても他 zip の処理を継続する（実 I/O 集計整合）

- **前提**
  - 実ファイル I/O で対象フォルダに複数 zip を配置する。
  - 少なくとも 1 件は成功可能な zip、1 件は失敗する zip（例: 画像 0 件）を含める。
- **操作**
  - ImportZipsService を実行する。
- **結果**
  - 失敗 zip が発生しても後続 zip の処理を継続する。
  - zip 単位結果一覧に成功・失敗が混在して記録される。
  - 全体サマリの件数が実 zip 件数と一致する。
  - `実行結果種別=PARTIAL_FAILURE` が返る。

---

### 同一入力では zip の処理順が安定する（実 FS + 実行再現性）

- **前提**
  - 実ファイル I/O で対象フォルダに、比較順が揺れやすい zip 名を混在配置する。
    - 例: `vol1.zip`, `vol02.zip`, `VOL2.zip`, `ＶＯＬ３.zip`, `vol10.zip`
  - 同一の入力フォルダ内容・設定で 2 回以上実行できる状態にする。
- **操作**
  - ImportZipsService を同一入力で複数回実行し、各回の zip 単位結果一覧の処理順を取得する。
- **結果**
  - すべての実行回で zip の処理順が一致する（非決定的な順序揺れがない）。
  - 並び順アルゴリズム自体の境界仕様（自然順/NFKC/大小文字）は small で担保済みとし、本ケースは「実 I/O 実行でも順序が再現すること」に責務を限定する。

---

### zip 単位でロールバックされ失敗 zip のゴミファイルが残らない

- **前提**
  - 実ファイル I/O で、展開途中で失敗する zip（例: 上限超過ファイルを含む）を用意する。
  - 成功する別 zip も同時に用意する。
- **操作**
  - ImportZipsService を実行する。
- **結果**
  - 失敗した zip に対応する一時展開物・中間生成物が作業領域に残らない。
  - 失敗 zip に対応するメディア永続化は行われない。
  - 成功 zip 側の永続化結果には影響しない。

---

### 上限超過入力は実行全体拒否となり副作用を残さない（最小統合ケース）

- **前提**
  - `1実行あたり最大zip処理件数=MAX_ZIPS_PER_RUN` を設定する。
  - 実ファイル I/O で zip 件数 `MAX_ZIPS_PER_RUN+1` の入力を用意する。
- **操作**
  - ImportZipsService を実行し、戻り値・永続化結果・作業領域を確認する。
- **結果**
  - `実行結果種別=INVALID_INPUT_RUN_ZIP_COUNT_LIMIT_EXCEEDED` を返す。
  - `zip単位結果一覧=[]` / `全体サマリ=0件` を返す。
  - zip ごとの処理が開始されず、永続化・展開作業の副作用を残さない。

> 補足: 上限 `-1 / = / +1` の境界値そのものは small で担保し、medium は「拒否時に実副作用がないこと」の統合確認に限定する。
