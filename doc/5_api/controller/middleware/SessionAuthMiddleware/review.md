# SessionAuthMiddleware 設計レビュー（過不足一覧）

## 結論
- 現状の設計は「Cookie名統一」「401レスポンス形式」「`request.context.userId` の基本契約」までは揃っている。
- 一方で、実装者ごとの差分やセキュリティ事故を防ぐための具体契約が不足している。

## 過不足一覧

### 1. `request.context` の初期化責務が未定義（不足）
- **現状**: `request.context.userId` を設定するとあるが、`request.context` が未定義の場合の扱いが書かれていない。
- **懸念**: 実装により `Cannot set property 'userId' of undefined` 相当の実行時エラーが起きうる。
- **補強案**: `request.context` が未作成ならミドルウェア側で生成してから `userId` を設定する、を明記する。

### 2. セッション有効性判定の観点が粗い（不足）
- **現状**: 「未指定・無効・期限切れ」で失敗とあるが、「無効」の判定条件が未定義。
- **懸念**: 署名不一致、失効済み、改ざん、存在しないセッションなどの扱いが実装依存になる。
- **補強案**: 最低限「形式不正 / 署名不一致 / ストア未存在 / 期限切れ / 明示失効」を失敗条件として列挙する。

### 3. セッションは有効でも `userId` 解決不能なケースが未定義（不足）
- **現状**: 成功時に `userId` を設定するとあるが、解決失敗時の扱いがない。
- **懸念**: 削除済みユーザー、壊れたセッションデータ時の返却コードが実装마다ぶれる。
- **補強案**: `userId` を導出できない場合は認証失敗として `401` を返す、と明記する。

### 4. 失敗時レスポンスのヘッダ契約が未定義（不足）
- **現状**: `401` + JSON本文は定義済み。
- **懸念**: キャッシュ制御や再認証誘導ヘッダの方針が揃わない。
- **補強案**: 必要なら `Cache-Control: no-store` 等の運用方針を追記する（不要なら「特別な追加ヘッダなし」と明記）。

### 5. ログ出力ポリシーが未定義（不足）
- **現状**: 監査ログ/アプリログの記載なし。
- **懸念**: 障害調査時に「なぜ401か」が追えない、または逆に機密を過剰記録する。
- **補強案**: `sessionId` 生値は出力しない、失敗理由コードのみ記録する等を明記する。

### 6. Cookie属性（Secure/HttpOnly/SameSite）の責務境界が曖昧（不足）
- **現状**: 検証対象Cookie名のみ定義。
- **懸念**: 認証基盤全体としての防御要件（CSRF/XSS耐性）が文書上で追跡しにくい。
- **補強案**: 本ミドルウェアの責務外なら「Cookie属性はログイン発行側設計を参照」とリンクで明示する。

### 7. 成功時の下流向け保証範囲は概ね妥当（充足）
- **現状**: `request.context.userId` が non-empty string である契約が明文化されている。
- **評価**: MediaPostControllerとのI/F整合は取れており、前回指摘は解消済み。

### 8. OpenAPIとのCookie名整合は達成（充足）
- **現状**: OpenAPI `cookieAuth.name` が `sessionId` に統一されている。
- **評価**: 命名不一致に起因する実装ブレの主要リスクは解消済み。

## 優先度付きの改善推奨
1. **高**: 項番1〜3（実装差分で不具合に直結）
2. **中**: 項番5（運用性・監査性）
3. **中**: 項番6（責務境界の明確化）
4. **低〜中**: 項番4（プロダクト方針次第）
